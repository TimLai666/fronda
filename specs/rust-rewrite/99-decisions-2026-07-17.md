一次定案 2026-07-17 audit 消化後的全部懸置決策。每項附選擇理由與「什麼情況下這個決策是錯的」。除 D6 外全部由本 session 定案並排入執行。

## D1. NDF-NTSC timecode 精度 — host seam，不動 media.json（EXECUTE）

**決策**：不在 manifest 加 frameDuration 欄位。改加 host seam `timecode_frame_duration(source) -> Option<(num, den)>`，ffmpeg 讀 tmcd stream 的 time_base，接進 `manifest_tc_seconds` 的換算（有值用精確值，無值維持 frame/quanta fallback）。
**理由**：Swift #269 正是同步時現場讀 CMTimeCodeFormatDescription，不存 manifest——鏡射架構同時保住 media.json 契約不分歧。
**錯誤條件**：若日後上游把 frameDuration 寫進 manifest，需改跟 on-disk 形。

## D2. Live audio 播放引擎 — cpal（EXECUTE，分階段）

**決策**：採 **cpal**（跨平台 audio I/O 事實標準）。不用 rodio（高階混音我們已有 audio_core mixer，rodio 疊床架屋）。架構：`app_shell_gpui::audio_playback` host adapter（platform adapter 規則），拉 audio_core 的 mix 流；階段一＝timeline 播放出聲＋live meter 餵真實輸出；階段二＝scrub audio（#293 全量）＋PCM cache（#339）。
**理由**：解鎖鏈最長（播放、meter、scrub、JKL 聲音）；cpal 無替代品級的競品。
**錯誤條件**：若 gpui-ce 未來內建 audio 層，應遷移以減依賴。

## D3. VAD — Silero via ONNX Runtime，optional feature（EXECUTE，先驗可建性）

**決策**：`vad` cargo feature ＋ `ort`（ONNX Runtime bindings）跑 Silero VAD（MIT，模型 ~2MB 可入庫），實作既有 `SpeechAnalyzer` seam；feature off 時行為不變（RMS fallback）。拒絕平台語音 API 路線（破壞跨平台一致性與確定性）。
**理由**：Silero 是上游 #261 的同款模型；seam 已就位，只差 host。
**錯誤條件**：ort 在三平台 CI 的建置成本若失控，退為 optional-by-default-off 並文件化；或改試 candle 重實作（工程大）。
**執行順序**：先 spike 驗 ort 在本機可建，再實作。
